# Hermes-PA Hotfixes — 2026-06-27 (apply to clones)

This is a portable playbook of every fix made on 2026-06-27. Apply it to **any
agent cloned from Hermes-PA before this date** — they all share these bugs.

All changes are on `Hermes-PA` `main` (PRs #1–#4). The fastest path for a clone
is to **merge these files from Hermes-PA**; the manual steps + verification below
let you apply them to a diverged clone and confirm each one on the live service.

Symptoms these fix: bot says **"model provider failure"** after attaching a
Railway volume; **memory/personas broken** after a rebuild; **no cron jobs** (or
**duplicated** cron jobs); **state wiped** when NousResearch builds a new image.

## Fix 1 — Railway volume shadows the baked config (the outage)

**Root cause.** `HERMES_HOME=/opt/data` and the image seeds config there. A
Railway **persistent volume mounts EMPTY and shadows** image content at its mount
path (Railway does NOT copy image data into a fresh volume, unlike Docker named
volumes). So `config.yaml` disappears → Hermes loses `model.provider=ollama-cloud`
and falls back to a default model Ollama Cloud doesn't host → **HTTP 404 /
"model provider failure"**. Adding `OLLAMA_API_KEY` doesn't help — the config that
*uses* it is gone. (Confirmed env var is correct: upstream `auth.py` →
`api_key_env_vars=("OLLAMA_API_KEY",)`.)

**Second bite.** A volume that already had `config.yaml` (e.g. attached before the
seed script existed) can be missing `profiles/`, `plugins/` (the Supabase memory
provider!), and `scripts/` → degraded memory, broken persona routing, no cron.

**Fix.** Bake an immutable seed *outside* the mount and copy it in on boot:

- `Dockerfile.railway`, in the Railway seed section:
  ```dockerfile
  COPY --from=source --chown=hermes:hermes /hermes/docker/railway-state/ /opt/data/
  COPY --from=source --chown=hermes:hermes /hermes/docker/railway-state/ /opt/railway-seed/
  COPY railway/seed-volume.sh /opt/hermes/seed-volume.sh
  RUN chmod 0755 /opt/hermes/seed-volume.sh && \
      printf '#!/command/with-contenv sh\nexec /opt/hermes/seed-volume.sh\n' > /etc/cont-init.d/00-railway-seed && \
      chmod +x /etc/cont-init.d/00-railway-seed
  ```
- `railway/seed-volume.sh`: on first boot (no `config.yaml`) copy the full seed in;
  on every boot **restore all config-as-code** — overwrite `config.yaml`+`SOUL.md`,
  **replace** `profiles/ plugins/ scripts/`, **merge** `skills/` — while never
  touching runtime state (`sessions/ cron/ kanban.db auth.json state.db memories/`).
  (Copy the file verbatim from Hermes-PA `main`.)

**Apply to a clone.** Copy `railway/seed-volume.sh`, apply the Dockerfile edits,
ensure the volume mount path is exactly **`/opt/data`**. No need to detach the
volume — on next deploy the script repairs it (seeds if empty, restores missing
dirs if partial).

## Fix 2 — CI/CD gate in front of the Railway auto-deploy

**Problem.** Pushes to `main` deployed straight to prod; nothing validated the
config or built/tested the image first. The old `deploy.yml` could even deploy an
unreviewed PR to the prod service.

**Fix.** `.github/workflows/ci.yml` with 4 jobs (PR + push to main):
- `config-lint` → `scripts/validate_config.py` (fails if `config.yaml` lacks
  `model.default` / `provider` / `base_url`, and if any profile has no `SOUL.md`).
- `unit-tests` → `pytest tests/` (incl. `tests/test_config_valid.py`).
- `secret-scan` → gitleaks (binary; no org license needed).
- `docker-smoke` → builds `Dockerfile.railway`, runs `seed-volume.sh` against an
  empty **Docker named volume**, asserts: first boot seeds config; second boot
  preserves runtime state; **hybrid boot restores deleted profiles/plugins/scripts**.
Plus `upstream-bump.yml` (weekly + manual) → opens a CI-tested PR bumping
`ARG HERMES_REF=` to the latest `NousResearch/hermes-agent` commit, so upstream
rebuilds become deliberate, not surprise wipes.
Delete `deploy.yml` and `tests.yml`.

**Deploy model:** Railway's **native GitHub auto-deploy** on `main` is the
deployer; the CI checks gate it. Lock it in with **branch protection on `main`**
requiring the 4 checks + "require a PR" (set in GitHub UI). Optional: Railway
service "Wait for CI"; repo secret `GH_PAT` so upstream-bump PRs auto-run CI.

**Apply to a clone.** Copy the 4 files, delete the 2, set branch protection. If a
clone uses a different Railway project/service name, update nothing in CI (CI
doesn't deploy) — just the Railway dashboard.

## Fix 3 — `setup-cron-jobs.py` was not idempotent (duplicate cron jobs)

**Root cause.** It checked existing jobs with `hermes cron list --json` — **there
is no `--json` flag**. The call errored, the check returned `[]`, so **every boot
recreated all jobs** → duplicates (e.g. 3× heartbeat firing every 15 min). The
"update" path used `hermes cron update`, also not a real subcommand (it's `edit`).

**Real cron commands:** `hermes cron list` (text only), `hermes cron create`,
`hermes cron remove <id>`, `hermes cron edit`. Jobs are stored at
`$HERMES_HOME/cron/jobs.json` → `{"jobs":[{ "id":..., "name":... }], ...}`.

**Fix.** `run_cron_list()` reads `jobs.json` directly; `main()` **skips** jobs that
already exist (create-only idempotency); removed the broken update path. Copy the
file from Hermes-PA `main`.

**Clean up existing duplicates** (one-time, per service — needs the prod shell):
```sh
railway ssh --service <SERVICE> 'sh -c "python3 -c \"
import json,subprocess
jobs=json.load(open(\\\"/opt/data/cron/jobs.json\\\")).get(\\\"jobs\\\",[])
seen=set(); dupes=[]
for j in jobs:
    n=j.get(\\\"name\\\");  dupes.append(j[\\\"id\\\"]) if n in seen else seen.add(n)
[subprocess.run([\\\"hermes\\\",\\\"cron\\\",\\\"remove\\\",i]) for i in dupes]
print(\\\"removed\\\", dupes)\" ; hermes cron list"'
```
(Keeps the first of each name, removes the rest. Run once per clone.)

## Fix 4 — Daily off-site backup to Supabase Storage

**Why.** Even with the volume, a volume loss = data loss. Memory is already in
Supabase, but `sessions/`, `kanban.db`, `cron/jobs.json`, `state.db` are not.

**Fix.** `scripts/backup.py` (stdlib only): tars **non-secret, non-regenerable**
state (`cron/jobs.json`, `kanban.db`, `kanban/`, `sessions/`, `state.db`,
`memories/`) and uploads to a private Supabase Storage bucket, pruning to the last
`HERMES_BACKUP_KEEP` (default 14). **Excludes** `.env`/`auth.json` (secrets,
rebuildable from Railway env) and config-as-code/caches (re-seeded from image).
Reads creds from env, falls back to `$HERMES_HOME/.env`. Telegram-alerts on
failure. Registered as no-agent cron `daily-backup` at `0 3 * * *` in
`setup-cron-jobs.py`, and added to the Dockerfile railway-state seed:
```dockerfile
COPY scripts/backup.py /hermes/docker/railway-state/scripts/backup.py
```
`.env.example` documents `HERMES_BACKUP_BUCKET` (default `hermes-backups`) and
`HERMES_BACKUP_KEEP` (default 14). Supabase `sb_secret_` keys work for the Storage
API. (Each clone gets its own bucket via the env var, or shares one — your call.)

**Apply to a clone.** Copy `scripts/backup.py`, add the cron entry + Dockerfile
COPY + `.env.example` vars. Verify with a one-off run after deploy:
```sh
railway ssh --service <SERVICE> 'python3 /opt/data/scripts/backup.py'
# -> backup: uploaded hermes-<ts>.tar.gz (NNN KB) to bucket 'hermes-backups', keep=14
```

## Deploy + verify (per clone)

1. Copy/merge the files above onto a branch → PR → wait for CI green → merge.
   Railway auto-builds `main` from the connected repo.
   - **Deploy local changes (no GitHub):** `railway up --service <SERVICE> --ci`
     (build streams). Do **not** use `--detach` for the first push — it can grab a
     stale cached build; `--ci` builds the real local tree.
2. Wait for the new deployment to be `RUNNING`, then verify (read-only):
   ```sh
   railway ssh --service <SERVICE> 'sh -c "
     for d in profiles plugins scripts skills; do printf \"%s: \" $d; [ -e /opt/data/$d ] && echo OK || echo MISSING; done
     ls /opt/data/plugins/supabase/__init__.py && echo PLUGIN_OK
     hermes cron list | grep -E \"Name:|Schedule:\"
   "'
   ```
   Expect: all dirs OK, `PLUGIN_OK`, and exactly one of each cron job.

## Operational gotchas (learned today)

- Volume mount path **must** be `/opt/data` (= `HERMES_HOME`).
- Ollama Cloud key env var is **`OLLAMA_API_KEY`** (+ `OLLAMA_BASE_URL=https://ollama.com/v1`).
- Changing **any** file copied into the image (config/scripts/etc.) busts the
  Docker `uv sync` cache layer → ~8-minute CI build. Expected.
- The boot order: cont-init runs `00-railway-seed` (restore dirs) → … →
  `03-railway-init` (writes git/Google creds, schedules `setup-cron-jobs.py` 30s
  after the gateway starts). Cron jobs therefore appear ~30s post-boot.
- Pushing to the default branch may be blocked by tooling policy — deploy via
  `railway up` or merge a PR instead.
